# readyHub Service Architecture

Этот документ описывает архитектуру frontend-сервиса readyHub и правила его дальнейшего расширения.

## 1. Продукт и рынок

readyHub работает на рынке B2B-решений для собственников и руководителей малого и среднего бизнеса.

Проблема рынка:

- Бизнесу сложно выбрать подходящие инструменты для роста, операционного порядка, команды, маркетинга и инвестиций.
- Решения часто разбросаны по консультантам, курсам, шаблонам и разрозненным базам знаний.
- Собственнику трудно понять, с чего начать и в какой последовательности внедрять изменения.

Суть продукта:

- Веб-платформа с каталогом готовых бизнес-решений.
- Решения объединяются в направления, готовые наборы, кейсы и статьи базы знаний.
- Пользователь может зайти через ситуацию, направление, поиск, набор или ручной подбор.
- Конечная цель интерфейса — помочь пользователю выбрать решение и оставить заявку.

Основная аудитория:

- Собственники SMB.
- Руководители отделов.
- Команды продукта, маркетинга, HR и операционного управления.
- Компании на стадиях `старт`, `рост`, `масштабирование`.

## 2. Технологический стек

Основной frontend:

- React 19.
- TypeScript strict.
- Vite 7.
- TanStack Router / TanStack Start.
- Tailwind CSS v4.
- Radix UI / shadcn-style components.
- lucide-react для иконок.
- clsx и tailwind-merge для className composition.

Подготовленные зависимости:

- React Query установлен и должен использоваться для будущей работы с API.
- react-hook-form и zod установлены и должны использоваться для будущих форм и валидации.
- Sonner установлен и может использоваться для toast-уведомлений.

Конфигурация:

- `vite.config.ts` использует `@lovable.dev/vite-tanstack-config`.
- Эта конфигурация уже включает React, Tailwind, TanStack Start, tsconfig paths и Cloudflare build-only.
- Не добавлять вручную дублирующие Vite-плагины в основной config.

## 3. Режимы сборки и публикации

Основная сборка:

- Команда: `npm run build`.
- Использует Vite/TanStack Start/Cloudflare-compatible pipeline.
- Предназначена для будущего SSR/Cloudflare-развёртывания.

Локальная разработка:

- Команда: `npm run dev`.
- Используется для разработки сайта.

GitHub Pages:

- Команда: `npm run build:gh-pages`.
- Использует `vite.github-pages.config.ts`.
- Пишет статическую сборку в `docs`.
- `src/main.github-pages.tsx` использует hash history, чтобы GitHub Pages корректно открывал вложенные маршруты.
- `docs/index.html`, `docs/404.html`, `docs/.nojekyll` должны оставаться частью публикации.

Правило:

- После UI/контентных изменений, которые должны попасть на GitHub Pages, обновлять `docs` через `npm run build:gh-pages`.
- Не редактировать файлы в `docs` вручную, кроме аварийных правок публикации.

## 4. Структура проекта

Основные директории:

- `src/routes` — файловые маршруты TanStack Router.
- `src/data` — текущие типы и статические данные каталога.
- `src/components/site` — доменные компоненты публичного сайта.
- `src/components/ui` — библиотечные shadcn/Radix UI-компоненты.
- `src/hooks` — общие React hooks.
- `src/lib` — общие утилиты.
- `src/styles.css` — Tailwind v4, дизайн-токены, базовые utilities.
- `docs` — статическая сборка для GitHub Pages.

Роли компонентов:

- `components/site` — компоненты, выражающие продукт readyHub: Header, Footer, Cards, CTA, FAQ, PageHero, Section.
- `components/ui` — низкоуровневые UI-примитивы. Их не нужно менять ради конкретной страницы, если можно решить задачу на уровне `components/site`.
- `routes` — страницы и route-level логика: loader, head meta, фильтры, локальное состояние страницы.
- `data` — пока временный источник truth для каталога и справочников.

## 5. Маршруты

Текущие публичные маршруты:

- `/` — главная страница.
- `/catalog` — каталог решений и наборов.
- `/catalog/$category` — страница направления.
- `/catalog/solution/$slug` — страница отдельного решения.
- `/catalog/bundle/$slug` — страница готового набора.
- `/cases` — кейсы.
- `/about` — о платформе.
- `/contacts` — заявка и контакты.
- `/blog` — база знаний.
- `/blog/$slug` — статья.
- `/privacy` — политика конфиденциальности.

Root layout:

- `src/routes/__root.tsx` задаёт Header, Footer, общий layout, default SEO и notFound.
- Все публичные страницы должны жить внутри этого layout.
- Для новых страниц обязательно задавать `head` meta: `title`, `description`, `og:title`, `og:description`.

Динамические маршруты:

- Должны использовать loader.
- Loader ищет сущность по `slug`.
- Если сущность не найдена, бросать `notFound()`.
- У каждой динамической страницы должен быть понятный `notFoundComponent`.

## 6. Архитектурные правила расширения

Добавление новой страницы:

1. Создать route-файл в `src/routes`.
2. Добавить `head` meta.
3. Использовать `PageHero` в начале страницы.
4. Использовать `Section`, `card-soft`, `FinalCTA` там, где это соответствует UI.
5. Если страница коммерческая или каталожная, завершить её `FinalCTA`.
6. Проверить mobile layout.

Добавление новой сущности каталога:

1. Обновить тип в `src/data/types.ts`, если нужны новые поля.
2. Добавить данные в `src/data/content.ts`.
3. Проверить связи по `slug`.
4. Проверить, что карточка и детальная страница используют новые данные без fallback-пустот.
5. Если сущность должна стать публичным маршрутом, добавить loader и notFound.

Добавление нового доменного компонента:

1. Размещать в `src/components/site`.
2. Не дублировать `PageHero`, `Section`, `FinalCTA`, `Cards`.
3. Использовать токены и utilities из `src/styles.css`.
4. Держать props простыми и типизированными.
5. Не смешивать data-fetching внутри презентационного компонента, если данные можно передать через props.

Добавление нового UI-примитива:

1. Размещать в `src/components/ui`.
2. Следовать shadcn/Radix-подходу.
3. Не завязывать UI-примитив на бизнес-термины readyHub.
4. Для доменной композиции создавать отдельный компонент в `components/site`.

## 7. Data flow сегодня

Сегодня каталог работает без backend:

- Данные импортируются из `src/data/content.ts`.
- Типы лежат в `src/data/types.ts`.
- Фильтрация каталога выполняется на клиенте через React state и `useMemo`.
- Динамические страницы получают данные через loader и поиск в статических массивах.
- Контактная форма показывает локальное состояние успешной отправки, но не отправляет данные на сервер.

Это допустимо для текущего статического сайта, но не должно считаться финальной backend-архитектурой.

## 8. Backend-вектор

Рекомендуемый будущий backend: Supabase.

Причины:

- Каталог имеет реляционные связи между категориями, решениями, наборами, статьями и кейсами.
- Нужны уникальные slug, внешние ключи и порядок сортировки.
- Нужна таблица заявок с контролируемой записью.
- В будущем возможна админка для управления каталогом и публикациями.

Firebase можно рассматривать только как альтернативу для быстрого прототипа, но базовый стандарт проекта — Supabase.

## 9. Правила качества

Перед сдачей изменений:

- `npm run lint` для проверки кода.
- `npm run build:gh-pages`, если изменения должны попасть на GitHub Pages.
- Проверить, что новые маршруты открываются напрямую.
- Проверить пустые состояния, notFound и мобильную сетку.
- Не коммитить случайные артефакты, кроме обновлённого `docs`, если публикация идёт через GitHub Pages.

## 10. Harness Engineering baseline

Для будущих агентов и разработчиков:

- Сначала читать `design.md`, `architecture.md`, `logic.md`.
- При UI-задачах следовать `design.md`.
- При маршрутах, структуре и сборке следовать `architecture.md`.
- При данных, backend, Supabase, API и валидации следовать `logic.md`.
- Если код противоречит документу, сначала понять, что актуальнее; затем обновить либо код, либо документ в рамках задачи.
